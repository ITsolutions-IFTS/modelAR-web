⚛️ React Native 2026

Introducción: El Panorama Completo

React Native en 2026 es radicalmente diferente a 2022. La arquitectura ha evolucionado con Bridgeless (JSI), Fabric (renderer), y mejor soporte para threading. Esta guía te explicará cómo funciona TODO, desde lo sincrónico hasta cómo se comunica con hardware.

🎯 Objetivo de esta guía: Entender por qué React Native moderno es 100-300x más rápido que versiones anteriores.

¿Qué aprenderás?

Concepto Por qué importa Impacto
Sync vs Async Base de todo en JS UI responsive vs congelada
Event Loop Cómo JS ejecuta código Entiende retrasos
Bridge vs JSI Comunicación con nativo 300x más rápido
Threading Paralelismo real App nunca se congela
Hardware CPU/GPU/Batería 15% más batería

El Roadmap Rápido

2019-2021: Bridge Viejo

Comunicación via JSON serialization + message queue. Lento (~200-500ms).

2022-2023: JSI (Bridgeless)

Acceso directo a memoria. Rápido (~1ms). Pero aún no era por defecto.

2024-2026: Bridgeless Default

JSI es el estándar. Fabric renderer. Hermes optimizado. React Native es moderno y rápido.

Sincronía vs Asincronía: El Corazón

JavaScript es single-threaded. Ejecuta código en UN hilo. Esto es fundamental para entender todo lo demás.

Sincrónico: Bloqueante

Sincronía: Esperas el resultado antes de continuar.
const sumar = (a, b) => { return a + b; // Respuesta inmediata }; const resultado = sumar(5, 3); console.log(resultado); // 8 // ⏱️ Tiempo total: <1ms

Asincrónico: No Bloqueante

Asincronía: Se ejecuta "después", sin bloquear.
const traerDatos = () => { return new Promise((resolve) => { setTimeout(() => { resolve({ id: 1 }); }, 1000); }); }; console.log("Empezando"); traerDatos(); console.log("Esperando..."); // Salida: // "Empezando" // "Esperando..." // (1s después) datos

Comparación Real

❌ Sincrónico Largo

  • Bloquea JS Thread
  • UI no se actualiza
  • Botones no responden
  • User: app "muerta"

✅ Asincrónico

  • JS Thread libre
  • UI responsiva
  • Botones funcionan
  • User: app fluida

Event Loop: El Maestro de Orquesta

Event Loop: Ejecuta código sincrónico primero, luego procesa callbacks cuando Call Stack vacío.

¿Cómo funciona?

┌─────────────────────────────────────┐ │ CALL STACK (ejecutándose) │ │ - console.log("1") │ │ - setTimeout(...) │ │ - console.log("3") │ └──────────────┬──────────────────────┘ ↓ (cuando está VACÍO) ┌─────────────────────────────────────┐ │ CALLBACK QUEUE (esperando) │ │ - callback del setTimeout() │ └──────────────┬──────────────────────┘ ↓ (Event Loop ejecuta) RESULTADO: "1" → "3" → (1s) "2"

Ejemplo Real

console.log("1"); setTimeout(() => { console.log("2"); }, 0); // Incluso con 0ms! console.log("3"); // RESULTADO: // 1 // 3 // 2 ← va al final (callback queue)

¿POR QUÉ Promises Se Ejecutan Primero?

Esta es la pregunta más importante. La respuesta está en HISTORIA de JavaScript:

1995-2008: Era setTimeout

setTimeout era la ÚNICA forma de diferir ejecución. No había "prioritarios".

2009: Nacen las Promises

Problema: Si Promise.then() espera a setTimeout, esto es LENTO:

const p1 = Promise.resolve().then(() => 'a'); const p2 = p1.then(() => 'b'); const p3 = p2.then(() => 'c'); // CON setTimeout (macro): 600-1500ms total! // p1 espera 200ms, p2 espera 200ms, p3 espera 200ms // CON microtasks: <1ms total ✅

Decisión: "Promises NECESITAN ir primero"

2011: MutationObserver

Observa cambios en DOM. "Ponlo con Promises" → Microtask

2013: El Estándar (ECMAScript)

DOS QUEUES: Microtask Queue (prioritario) + Macrotask Queue (normal)

¿QUÉ Determina Micro vs Macro?

MICROTASKS (Prioritario) MACROTASKS (Normal) SPECIAL
Promise.then() setTimeout() requestAnimationFrame()
Promise.catch() setInterval() (Ejecuta ANTES de repaint)
Promise.finally() setImmediate() (Node.js)
queueMicrotask() fetch() (I/O)
MutationObserver Eventos (click, input)
async/await (es Promise debajo)

El Algoritmo Real del Event Loop

while (eventLoop.waitForTask()) { // 1. Ejecuta UNA macrotask const macrotask = macrotaskQueue.pop(); execute(macrotask); // 2. EJECUTA TODAS las microtasks while (microtaskQueue.hasTasks()) { const microtask = microtaskQueue.pop(); execute(microtask); } // 3. Si hay cambios, renderiza if (isRepaintTime()) { repaint(); } } // CLAVE: // - UNA macrotask por ciclo // - TODAS las microtasks por ciclo ← POR ESO PROMISES PRIMERO

Ejemplo Real: Orden de Ejecución

console.log("1"); // Sincrónico setTimeout(() => { console.log("2"); // MACROTASK }, 0); Promise.resolve() .then(() => console.log("3")) // MICROTASK .catch(() => console.log("4")) // MICROTASK .finally(() => console.log("5")); // MICROTASK queueMicrotask(() => { console.log("6"); // MICROTASK }); const observer = new MutationObserver(() => { console.log("7"); // MICROTASK }); observer.observe(document.body, { attributes: true }); document.body.setAttribute('data', 'changed'); requestAnimationFrame(() => { console.log("8"); // SPECIAL (antes de repaint) }); setInterval(() => { console.log("9"); // MACROTASK }, 0); console.log("10"); // Sincrónico // RESULTADO: // 1 ← Sincrónico (primero) // 10 ← Sincrónico (primero) // 3 ← Promise.then (MICROTASK) // 4 ← Promise.catch (MICROTASK) // 5 ← Promise.finally (MICROTASK) // 6 ← queueMicrotask (MICROTASK) // 7 ← MutationObserver (MICROTASK) // 8 ← requestAnimationFrame (SPECIAL) // 2 ← setTimeout (MACROTASK) // 9 ← setInterval (MACROTASK)

¿Por Qué Este Orden? (3 Razones Técnicas)

1️⃣ Promises Cadenas Largas

Sin microtasks: Cada .then() espera 200ms

p1→p2→p3 = 600ms ❌

Con microtasks = <1ms ✅

2️⃣ DOM Mutations

MutationObserver DEBE verse en siguiente repaint

Si espera a setTimeout, DOM se ve viejo

3️⃣ Rendering Budget (16.67ms)

Frame: ├─ Sync code (2ms) ├─ ALL Microtasks (3ms) ← ANTES de repaint ├─ ONE Macrotask (4ms) ← DESPUÉS └─ Render (6ms) Si Promises esperan a setTimeout: ├─ Sync ├─ setTimeout #1 ├─ AHORA ejecutan Promises (TOO LATE) └─ Render ya ocurrió (con datos viejos)

Por Qué Importa en React Native

Promises (Microtasks)

Ejecutan rapidísimo después que el Stack vacía

NO bloquean UI Thread

Ideales para operaciones async rápidas

setTimeout/fetch (Macrotasks)

Ejecutan después de TODAS las Microtasks

Perfectas para operaciones I/O lentas

Respetan el Event Loop

🔑 Importante:
Event Loop es lo que permite UI responsiva. Promises (Microtasks) ejecutan casi instantáneamente, mientras que setTimeout/fetch (Macrotasks) respetan la cola. Usa Promises para cosas rápidas, Macrotasks para I/O.

Bridge Viejo: El Bus de Comunicación Lento

El Bridge es el mecanismo de comunicación entre JavaScript y el código nativo (Objective-C en iOS, Java en Android). Hasta 2023, era la ÚNICA forma de comunicarse. Y era LENTO.

¿Por Qué Existen dos Mundos?

JavaScript (C++ Hermes Engine): Corre en su propio "proceso virtual" aislado, con su propia memoria.

Código Nativo (Objective-C/Java): Corre directamente en el OS, tiene acceso a hardware real (cámara, GPS, sensores).

El Problema: No comparten memoria. No pueden accederse directamente uno a otro. Necesitan un "bus de comunicación".

El Bridge: Una Cola de Mensajes

El Bridge es literalmente una cola (queue) que pasa mensajes entre JS y Native, transformando datos en JSON.

┌─────────────────────────────────────────────────────┐ │ JAVASCRIPT THREAD (Hermes C++) │ │ │ │ const user = { id: 1, name: "Juan" }; │ │ nativeModule.saveUser(user); // ← Llamada │ │ │ │ (JavaScript espera aquí - Promise PENDIENTE) │ └────────────┬────────────────────────────────────────┘ │ ↓ [BRIDGE - Cola de Mensajes] │ ┌──────────────────┐ │ MESSAGE QUEUE │ │ ┌──────────────┐ │ │ │ method: "save" │ │ data: "{...}"│ │ ← JSON serializado │ │ id: 123 │ │ ← ID para volver │ └──────────────┘ │ └────────┬─────────┘ │ ↓ [NATIVE THREAD procesa - toma ~10-50ms] │ ┌──────────────┴──────────────────────────────────────┐ │ NATIVE THREAD (iOS Objective-C) │ │ │ │ @objc(saveUser:) │ │ func saveUser(_ userDict: [String: Any]) { │ │ // 1. Recibe dict de la queue │ │ // 2. Valida y procesa │ │ // 3. Accede a BD, archivos, etc. │ │ // 4. Prepara respuesta │ │ } │ └────────────┬─────────────────────────────────────────┘ │ ↓ [Respuesta serializa A JSON - OTRA VEZ] │ ┌──────────────────┐ │ RESPUESTA QUEUE │ │ ┌──────────────┐ │ │ │ id: 123 │ │ ← ID para matching │ │ result: "{...}" ││ │ └──────────────┘ │ └────────┬─────────┘ │ ↓ [200-400ms ESPERANDO] │ ┌──────────────┴──────────────────────────────────────┐ │ JAVASCRIPT THREAD (sigue esperando) │ │ │ │ const user = await nativeModule.saveUser(user); │ │ ↑ FINALMENTE LA RESPUESTA LLEGA │ │ ↑ Event Loop ejecuta el callback de la Promise │ └──────────────────────────────────────────────────────┘

El Flujo Paso a Paso: Cargar Foto de Galería

t=0ms: Usuario presiona "Elegir Foto"

JS llama: nativeModule.pickImage()

t=1-2ms: Serialización

JS convierte datos a JSON string. Esto toma CPU.

// JS objeto: { options: { width: 300, height: 300 }, format: "jpeg" } // Serializa a: "{\"options\":{\"width\":300,\"height\":300},\"format\":\"jpeg\"}"
t=3-5ms: Cola de mensajes

El mensaje entra en la Bridge queue, esperando a ser procesado por el Native Thread.

t=6-50ms: Native procesa

iOS/Android recibe el JSON, parsea, abre la galería de fotos.

// Objective-C (iOS) @objc(pickImage:) func pickImage(_ options: NSDictionary) { let picker = UIImagePickerController() // Abre galería self.present(picker, animated: true) }
t=50-200ms: Usuario elige foto (interacción nativa)

El Native thread procesa mientras JS espera. Promise PENDIENTE.

t=200-202ms: Serialización respuesta

Native convierte resultado a JSON string nuevamente.

// Respuesta (en JSON): "{\"uri\":\"file://...photo.jpg\",\"width\":300,\"height\":300}"
t=203-205ms: Cola de respuesta

Respuesta entra en otra queue, esperando que JS la procese.

t=206-210ms: Desserialización

JS parsea JSON string de vuelta a objeto JavaScript.

t=211ms: Promise resuelve

Event Loop ejecuta el callback. Usuario ve foto hace ~210ms de retraso SIN RAZÓN TÉCNICA.

El Problema: Serialización (Traducir Toma Tiempo)

La serialización es convertir un objeto de memoria a un string de texto. Es como traducir de español a código morse: posible, pero tedioso.

❌ Lo que DEBERÍAS hacer (rápido)

// Apunta directamente a memoria const photo = getNativePhoto(); // <1ms console.log(photo.uri); // Acceso directo

❌ Lo que Bridge HACE (lento)

// Convierte a JSON string const photo = await nativeModule.getPhoto(); // 1. JS serializa request a JSON (~1ms) // 2. Envía por Bridge (~2ms) // 3. Native parsea JSON (~1ms) // 4. Procesa (~50ms) // 5. Serializa respuesta (~1ms) // 6. Envía de vuelta (~2ms) // 7. JS desserializa (~1ms) // TOTAL: ~58ms + esperas en queues // REAL: 200-500ms

Números Reales de Overhead

Operación Tiempo Real Overhead Por qué
Serialización (→ JSON) 1-2ms 50-100% Convertir objeto a string
Envío por Bridge 2-5ms Variável Depende de carga
Native procesa 10-50ms 0% (trabajo real) Acceso a BD, cámara, etc.
Desserialización (← JSON) 1-2ms 50-100% Convertir string a objeto
Event Loop callback 1-5ms Variável Espera a que Stack vacíe
TOTAL ~200-500ms ~80% OVERHEAD Traducción ineficiente

Analogía: El Teléfono Descompuesto

JavaScript: Persona en habitación 1 (no puede salir)

Native: Persona en habitación 2 (no puede entrar)

Bridge: Empleado que lleva mensajes entre habitaciones, PERO solo entiende Morse.

El Problema:

  • 1. Persona 1 habla español, empleado lo traduce a Morse (~1ms)
  • 2. Empleado camina a habitación 2 (~2ms)
  • 3. Persona 2 recibe Morse, lo traduce a español (~1ms)
  • 4. Persona 2 responde algo (~50ms de trabajo real)
  • 5. Persona 2 traduce respuesta a Morse (~1ms)
  • 6. Empleado camina de vuelta (~2ms)
  • 7. Persona 1 traduce Morse de vuelta a español (~1ms)
  • TOTAL: ~210ms + esperando al empleado
🔑 La Raíz del Problema:
• JS y Native son dos "habitaciones" de memoria separadas
• No pueden accederse directamente
• Necesitan traducir a JSON (como Morse)
• Traducir toma ~80% del tiempo total
• Esto es INEFICIENTE para operaciones frecuentes

Solución: JSI permite que compartan la MISMA memoria, sin traducción.

JSI Bridgeless: Acceso Directo a Memoria

JSI (JavaScript Interface) es lo opuesto al Bridge. En lugar de traducir y pasar mensajes, JSI permite que JavaScript y Native compartan la MISMA memoria. Es como si las dos "habitaciones" fueran la misma.

JSI: C++ layer que permite que JS acceda DIRECTAMENTE a objetos nativos en memoria, sin serialización.

¿Cómo Funciona? La Magia de C++

Tanto JavaScript (Hermes) como el código Nativo (iOS/Android) pueden compilarse a C++. JSI es una "zona" de C++ compartida donde ambos pueden ver la misma memoria.

┌──────────────────────────────────────────────────────┐ │ MEMORIA COMPARTIDA (C++) │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ Objeto User en MEMORIA REAL │ │ │ │ { │ │ │ │ id: 1 │ │ │ │ name: "Juan" │ │ │ │ email: "juan@example.com" │ │ │ │ } │ │ │ │ │ │ │ │ ↑ Ambos apuntan DIRECTAMENTE │ │ │ │ No hay copias, no hay traducción │ │ │ └─────────────────────────────────────────┘ │ │ ↑ ↑ │ │ │ │ │ │ ┌───────────┐ ┌──────────────────┐ │ │ │ JS Thread │ │ Native Thread │ │ │ │ │ │ │ │ │ │ user.name │ │ user->getName() │ │ │ │ (acceso) │ │ (acceso) │ │ │ └───────────┘ └──────────────────┘ │ └──────────────────────────────────────────────────────┘

Ejemplo Real: Acceso Directo

Con JSI, JavaScript puede llamar métodos nativos de forma síncrona y obtener respuestas instantáneamente.

Con Bridge (LENTO):
const getPhotoCount = async () => { // 1. Serializa → JSON // 2. Envía por queue // 3. Native procesa // 4. Serializa respuesta // 5. Desserializa const count = await nativeModule.getPhotoCount(); // ~200ms ESPERA };
Con JSI (RÁPIDO - Sincrónico):
const getPhotoCount = () => { // Acceso DIRECTO a memoria // Sin serialización // Respuesta INMEDIATA const count = nativeModule.getPhotoCount(); // <1ms TOTAL return count; };

TurboModules: Módulos Rápidos con JSI

TurboModules son módulos nativos que usan JSI. Se generan automáticamente con Codegen.

Definir un TurboModule (TypeScript):

// TypeScript especifica la interfaz export interface Spec extends TurboModule { getPhotoCount(): number; savePhoto(path: string): boolean; pickImage(): Promise<{ uri: string }>; } export default TurboModuleRegistry.getEnforcing<Spec>( 'PhotoModule' );

Implementar en Objective-C (iOS):

@objc(PhotoModule) class PhotoModule: NSObject, PhotoModuleSpec { @objc static func moduleName() -> String! { return "PhotoModule" } // JSI: Acceso directo, sin serialización @objc func getPhotoCount() -> NSNumber { let count = PHPhotoLibrary.authorizationStatus() return NSNumber(value: count) // Directo } // Puede ser sincrónico gracias a JSI @objc func savePhoto(path: String) -> NSNumber { let saved = UIImage(contentsOfFile: path) return NSNumber(value: saved != nil) } // Operaciones lentas aún pueden ser async @objc func pickImage() -> AnyPromise { return Promise { picker in let picker = UIImagePickerController() // ... } } }

Comparación: Bridge vs JSI

Aspecto ❌ Bridge ✅ JSI
Comunicación Colas asincrónicas (obligatorio) Sincrónica posible (directo)
Serialización JSON (4-6 veces por operación) Ninguna (acceso directo a C++)
Tiempo ~200-500ms <1ms
Copias de datos 4-6 copias en memoria 0 copias (misma memoria)
CPU Usage Alto (serializar/desserializar) Bajo (acceso directo)
Batería Consumo alto Consumo bajo (10-15% ahorro)
VELOCIDAD RELATIVA 1x (baseline lento) 300x (instantáneo)

¿Pero Por Qué Aún Existe el Bridge?

Aunque JSI es mejor, el Bridge aún existe porque:

  • Compatibilidad: Módulos viejos aún usan Bridge
  • Seguridad: Bridge aísla mejor (JS no puede corromper Native)
  • Migración: No todos los módulos se han actualizado a JSI

Pero para código nuevo (2022+): Usa siempre JSI/TurboModules.

🔑 La Revolución de JSI:
• 300x más rápido que Bridge
• Sin serialización: Acceso directo a C++ en memoria
• Sincronía posible: Llamadas que devuelven respuesta instantáneamente
• Menos CPU: Sin overhead de traducción
• Más batería: 10-15% ahorro real
• Type-safe: Codegen genera glue code automáticamente

Conclusión: JSI Bridgeless es la razón por la que React Native moderno es viable para apps demandantes (chat, redes sociales, streaming).

Threading: Cómo React Native Corre en Paralelo

React Native no ejecuta TODO en un solo hilo. En realidad usa 4 threads independientes que se coordinan para hacer que la app sea responsive. Entender cómo se coordinan es la clave.

Thread: Camino de ejecución independiente. Un dispositivo multi-core puede ejecutar múltiples threads EN PARALELO en diferentes cores del CPU.

Los 4 Threads Vitales

1️⃣ UI Thread (Main Thread) - EL MÁS CRÍTICO

¿Qué es? El único hilo que el Sistema Operativo permite tocar la UI nativa (UIView en iOS, View en Android).

Por qué es crítico: Cada 16.67ms (para 60fps), el OS pregunta: "¿Hay algo nuevo que pintar?" Si el UI Thread está ocupado, el OS no puede actualizar la pantalla → el usuario ve "congelamiento".

Budget: 16.67ms para TODA la lógica de renderizado. Si usas 20ms, pierdes 1 frame.

// ❌ PROHIBIDO - Acceder UI desde JS Thread const btn = document.getElementById("myBtn"); btn.style.backgroundColor = "red"; // Error: Acceso desde thread incorrecto! // ✅ CORRECTO - Usar runOnUiThread import { runOnUiThread } from 'react-native'; runOnUiThread(() => { // Ahora estamos en UI Thread btn.style.backgroundColor = "red"; // OK });

2️⃣ JavaScript Thread - El Cerebro

¿Qué es? Donde corre TODOS tus archivos JS/TS. React, lógica de app, Event Loop, todo.

Motor: Hermes (interprete C++) que ejecuta tu código JavaScript compilado a bytecode.

El peligro: Si haces un ciclo largo AQUÍ, bloquea todo el JS Thread, y aunque UI Thread esté libre, no puede recibir órdenes de React.

❌ BLOQUEA JS Thread

const handlePress = () => { // Ciclo ENORME en JS Thread let sum = 0; for (let i = 0; i < 1000000000; i++) { sum += i; // 5 segundos bloqueado } // Durante estos 5 segundos: // - React NO puede procesar eventos // - UI NO se actualiza // - Botones NO responden // - Usuario ve app "muerta" };

✅ DELEGA a Worker

import { useSharedValue, runOnJS } from 'react-native-reanimated'; const handlePress = async () => { // Delega a NATIVE THREAD const result = await spawnWorker(() => { let sum = 0; for (let i = 0; i < 1e9; i++) { sum += i; } return sum; }); // Durante estos 5 segundos: // - JS Thread LIBRE // - React PUEDE procesar eventos // - UI SE ACTUALIZA // - Botones RESPONDEN // - Usuario ve app normal };

3️⃣ Render Thread (Android L+)

Thread especializado en renderizar UI con OpenGL. Es lo que dibuja realmente los píxeles. Con Fabric, es más eficiente.

4️⃣ Native Modules Thread

Para operaciones nativas que toman tiempo (cámara, BD, red). JS no espera bloqueado, sino con Promises/async-await.

Cómo se Coordinan: Una Llamada Real

Veamos qué pasa cuando el usuario toca un botón hasta que ve el cambio en pantalla:

t=0ms: Touch event en pantalla

Thread: UI Thread

OS captura el toque del usuario

t=1ms: React.press callback

Thread: JS Thread

UI Thread NOTIFICA al JS Thread que hubo un toque

<Button onPress={() => { // Esto se ejecuta en JS Thread setCount(c => c + 1); }} />
t=2-3ms: React.render (diffa)

Thread: JS Thread

React calcula qué cambió en Virtual DOM

t=4-5ms: Fabric Layout (Yoga)

Thread: JS Thread → Fabric (C++)

Calcula dimensiones y posiciones con JSI

t=6-10ms: GPU Draw

Thread: Render Thread + GPU

Render Thread instruye a GPU que dibuje usando OpenGL

t=11-15ms: Frame buffer ready

Thread: UI Thread

Core Animation (iOS) o RenderThread (Android) prepara el siguiente frame

t=16-17ms: VSYNC - Pantalla actualiza

Resultado: ✅ Usuario ve el cambio. 60fps = éxito

El Budget Crítico: 16.67ms

Para mantener 60fps (1000ms ÷ 60 frames = 16.67ms por frame), TODO debe ocurrir en menos de 16.67ms:

Fase Thread Tiempo ideal Si se pasa
React diffa JS Thread 2-3ms React es lento → lag
Fabric layout JS Thread 2-3ms Layout es lento → lag
GPU draw Render + GPU 4-6ms GPU no termina → frame perdido
UI update UI Thread 1-2ms Core Anim toma mucho → frame perdido
TOTAL - < 16.67ms Si > 16.67ms → perdés 1 frame

Visualización: Qué Pasa si Alguien Toma Demasiado Tiempo

BUEN CASO (60fps): ┌────────────────────────────────────────┐ │ Frame 1: React (3ms) + Fabric (3ms) │ │ + GPU (6ms) + UI (2ms) = 14ms │ │ ✅ Cabe en 16.67ms │ └────────────────────────────────────────┘ MALO - React toma mucho (30ms): ┌────────────────────────────────────────┐ │ React (30ms) >>> 16.67ms LÍMITE ❌ │ │ GPU y UI Thread esperan a que │ │ JS Thread termine │ │ Resultado: Frame se salta, usuario ve │ │ "congelamiento" 33ms de retraso │ └────────────────────────────────────────┘ MALO - GPU toma mucho: ┌────────────────────────────────────────┐ │ React (3ms) + Fabric (3ms) OK │ │ GPU (25ms) >>> 16.67ms ❌ │ │ GPU no termina a tiempo │ │ Frame se salta, usuario ve lag │ └────────────────────────────────────────┘
🔑 La Regla de Oro del Threading:
• UI Thread: NUNCA > 16.67ms. Si necesitas más, rompes 60fps.
• JS Thread: NUNCA > 16.67ms. Si necesitas más, delega a Worker.
• Render Thread: NUNCA > 16.67ms. GPU debe terminar para VSYNC.
• Worker Threads: Pueden tomar lo que necesiten, no bloquean UI.

Conclusión: Threading no es sobre "hacer 4 cosas a la vez". Es sobre coordinación perfecta de 4 actores en 16.67ms para que el usuario vea transiciones suave. Si uno falla, todo falla.

Hardware: CPU, GPU, Batería y ARM

React Native corre en smartphones con limitaciones de recursos REALES. Entender cómo el código impacta hardware es fundamental.

CPU: El Procesador Principal

CPU: Ejecuta instrucciones de código. Cada instrucción consume energía eléctrica.

Concepto clave: El CPU es como una máquina que puede estar a diferentes velocidades. Más instrucciones = más rápido = más batería.

Estado CPU Consumo Corriente Duración Batería Qué está pasando
100% (máximo) 500-800mA ~5 horas Compilando, procesando, cálculos
50% (normal) 250-400mA ~10 horas App corriendo, respuesta normal
10% (idle) 50-100mA ~30 horas Esperando entrada, muy poco trabajo
Dormido (sleep) <10mA >100 horas Pantalla apagada, app en background

La conexión con React Native: Si tu código hace muchas operaciones sin necesidad (serializar JSON, polling frecuente), CPU está constantemente a 100%. Resultado: batería muere en 5 horas.

❌ CONSUMO ALTO

// Polling cada 100ms setInterval(() => { checkLocation(); // Despierta CPU }, 100); // 600 veces por minuto // CPU nunca puede dormir // Batería drena en 4-5 horas

✅ CONSUMO BAJO

// Polling cada 5 minutos setInterval(() => { checkLocation(); // Despierta CPU }, 5 * 60 * 1000); // 12 veces por minuto // CPU puede dormir 5 minutos // Batería dura 15+ horas

GPU: El Procesador Gráfico Especializado

GPU: Procesador especializado en dibujar. Paralelo masivo (miles de cores), pero solo sabe hacer gráficos.

La diferencia: CPU dibuja 1 pixel por ciclo. GPU dibuja 1000 píxeles en paralelo. Es como comparar un pintor (CPU) vs 1000 pintores (GPU).

En React Native: Fabric (el renderer nuevo) delega TODO a GPU en lugar de CPU.

Resultado: GPU está optimizada para esto, no interfiere con CPU que puede hacer lógica.

Operación CPU GPU (Fabric) Mejora
Medir 5000 textos ~300ms ~240ms 20% más rápido
Animar scroll 30-40fps 60fps 2x más fluido
Transformaciones (rotate, scale) Lento Instantáneo 100x más rápido
Memoria GPU N/A 25% menos Más espacio para app

Batería: Cómo Todo Se Convierte en Calor

La batería no "se gasta". Se convierte en energía eléctrica que el CPU/GPU consume. Esa energía se disipa como CALOR.

Batería (3000-4500 mAh típica) ↓ Regulator (entrega ~3.7V) ↓ ┌─────────────────────────────────────┐ │ CPU (50% del consumo) │ ├─────────────────────────────────────┤ │ GPU (20% del consumo) │ ├─────────────────────────────────────┤ │ Pantalla (20% del consumo) │ ├─────────────────────────────────────┤ │ Radio (WiFi/4G) (10% del consumo) │ └─────────────────────────────────────┘ ↓ Se disipa como CALOR ↓ Teléfono se calienta ↓ Batería se agota EJEMPLO DE DURACIÓN: - Standby: >100 horas - 2G/3G talk: 10-15 horas - Video playback: 8-10 horas - Web browsing: 5-8 horas - React Native app pesada: 4-6 horas

¿Cómo Bridgeless Ahorra Batería?

JSI Bridgeless ahorra 10-15% de batería porque NO hace serialización:

❌ Bridge: Mucho Trabajo CPU

Serializar JSON: CPU 100%

Enviar por bridge: CPU 50%

Desserializar JSON: CPU 100%

Resultado: CPU constantemente trabajando, batería se drena rápido

✅ JSI: Acceso Directo

Acceso a memoria: CPU 5%

Respuesta directa: CPU 5%

Sin traducción: Sin overhead

Resultado: CPU hace mínimo trabajo, batería dura más

ARM: La Arquitectura del Smartphone

ARM: Arquitectura de procesador diseñada específicamente para dispositivos móviles. Enfatiza eficiencia sobre velocidad pura.
Característica ARM (Móvil) x86 (PC) Por qué
Filosofía Eficiencia primero Velocidad primero ARM: batería limitada. x86: enchufado
Consumo batería Bajo (50-800mA) Alto (30-100W) ARM ahorra energía por instrucción
Clock speed 1.5-3.5 GHz 2.5-5.0 GHz Menos velocidad = menos calor
Cores 4-8 (heterogéneo) 4-16 (homogéneo) ARM mezcla cores rápidos + lentos
Instrucciones Simples (32/16 bits) Complejas (x86 variable) ARM: más instrucciones, menos energía

🔑 La Conexión: Bridgeless + ARM
• ARM está optimizado para eficiencia (no desperdiciar batería)
• Bridgeless respeta esa eficiencia (acceso directo, sin traducción)
• Resultado: 10-15% ahorro de batería real
• En práctica: App dura 2 horas más con misma batería

Por eso las grandes empresas (WhatsApp, Instagram, Discord) usan React Native moderno (con JSI) - la batería importa.

Hermes: El Motor JavaScript Optimizado para Mobile

Hermes es el motor JavaScript creado por Meta (Facebook) específicamente para React Native. No es solo un JavaScript interpreter - es una máquina virtual OPTIMIZADA para teléfonos.

Hermes: Motor JavaScript que compila código a bytecode (forma intermedia) antes de ejecutar. Diseñado específicamente para minimizar RAM y maximizar startup time en dispositivos móviles.

¿Por Qué No Usar V8 (Chrome)?

V8 es excelente para navegadores, pero está OPTIMIZADO para PCs con mucha RAM y CPU potente. En un teléfono, es un desperdicio.

Métrica V8 (Chrome) Hermes Impacto
Tamaño del engine 8-15MB 3-4MB APK 75% más pequeño
RAM al startup 50-80MB 30-40MB Deja más RAM libre para app
Tiempo para botar 400-600ms 200-300ms App aparece 2x más rápido
JIT Compilation Sí (overhead inicial) No (interpretado directo) Startup más predecible
Memory leaks Más probables Menos probables Mejor para long-running apps

Cómo Funciona Hermes: De Código a Ejecución

Cuando buildeás una React Native app con Hermes, el flujo es:

1. Escribes Código JavaScript
const calculateSum = () => { let total = 0; for (let i = 1; i <= 100; i++) { total += i; } return total; }; calculateSum();
2. Hermes Parser (Análisis)

Lee el código, valida sintaxis, crea AST (Abstract Syntax Tree)

3. Compilación a Bytecode

Convierte AST a HBC (Hermes ByteCode) - formato binario optimizado

// Bytecode (binario): // 0x0A 0x15 0xF2 0x3C ... (secuencia binaria) // Mucho más pequeño que JavaScript texto // Más rápido de parsear
4. Bundling para APK/IPA

El bytecode se empaqueta en el APK/IPA de tu app

5. User Instala App

APK es más pequeño (gracias a Hermes)

Descarga: 5 minutos vs 20 minutos con V8

6. App Startup

Hermes interpreter carga bytecode en RAM (30-40MB)

VS V8 que necesita 50-80MB

7. Hermes Interpreter (Ejecución)

Interpreta bytecode como máquina virtual

Una instrucción por ciclo (predecible)

8. Usuario ve app en pantalla

~200-300ms con Hermes

vs ~400-600ms con V8

Bytecode: La Magia de Hermes

Bytecode es el "idioma intermedio" entre JavaScript (texto) y máquina (binario). Es como una "forma condensada".

❌ V8: Texto Crudo

// Archivo .js puro const x = 5; const y = 10; return x + y; // Startup: // 1. Leer archivo (~2ms) // 2. Parsear texto (~50ms) // 3. JIT compilar (~200ms) // 4. Ejecutar // TOTAL: ~250ms + JIT pauses

✅ Hermes: Bytecode Compilado

// Archivo .hbc (bytecode) // 0x0F 0x2A 0x10 ... (binario) // Startup: // 1. Leer bytecode (~1ms) // 2. Cargar en memoria (~20ms) // 3. Ejecutar (sin compilar) // TOTAL: ~25ms (sin JIT pauses)

La Realidad: Benchmarks Reales

Escenario V8 Hermes Diferencia

Warm startup (desde cache) 400ms 150ms ⚡ 2.6x más rápido
APK size +15MB engine +3MB engine 📦 80% más pequeño
RAM usage al boot 80MB 35MB 🧠 56% menos RAM
Garbage Collection pauses Frequent Less frequent ✅ Menos jank
🔑 Por qué Hermes importa en React Native:
• APK 75% más pequeño: Menos data para descargar, instalar
• Startup 2x más rápido: App aparece en 300ms vs 600ms
• Menos RAM: 40% menos, más espacio para usuario
• Sin JIT overhead: Startup predecible, sin pauses
• Diseñado para móvil: No desperdicia ciclos en features de PC

Conclusión: Hermes es POR QUÉ React Native moderno puede competir con apps nativas. Sin Hermes, React Native apps serían 2x más lentas al abrir.

Best Practices: React Native Rápido

1. Usa Async/Await (No Callbacks Anidados)

❌ Callbacks anidados

fetchUser("id", (err, user) => { fetchPosts(user.id, (err, posts) => { // Nivel 3 anidamiento }); });

✅ Async/Await

const load = async () => { const user = await fetchUser("id"); const posts = await fetchPosts( user.id ); };

2. Nunca Bloquees UI Thread (> 16ms)

Si una operación tarda > 16ms, delégala a thread separado.

3. Memoriza Componentes (useMemo, useCallback)

Evita re-renders innecesarios = menos JS trabajo = menos lag.

4. Usa React.memo

Solo re-renderiza si props cambian, no si parent se actualiza.

5. Optimiza FlatList (virtualización)

Solo renderiza items visibles, no todos.

6. Monitorea con React DevTools Profiler

Ve exactamente qué es lento y por qué.

7. Usa Fabric + JSI (RN 0.73+)

Actualiza a React Native 0.73 o superior para Bridgeless automático.

8. Gestiona Batería Inteligentemente

// ❌ Drenador: cada 100ms setInterval(() => { checkLocation(); }, 100); // ✅ Ahorrativo: cada 5 minutos setInterval(() => { checkLocation(); }, 300000);

Escribe código que respete hardware. 60fps, batería, CPU - todo es tradeoff. Entiende Event Loop, respeta 16.67ms budget, delega trabajo pesado a threads separados.
